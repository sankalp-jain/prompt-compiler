## What was built

### Meta-prompt engineering
- Introduced **intent summarisation step** (internal, not in output): goal, domain, constraints, depth
- Enforced **5-section system prompt structure**: Role & expertise, Task objective, Constraints & assumptions, Required output structure, Quality bar
- Split output into 6 JSON fields assembled in code — structure enforced by schema, not model behaviour
- Added **Reviewer Prompt** (fixed, always appended): senior architect review checklist
- Added **quality bar rules** word-for-word: senior reader, explicit decisions, trade-off justification, no filler
- Identified and fixed 6 failure cases: stack invention, non-constraining bullets, weak task objective, underspecified sections, redundant user prompt, missing use-case quality criteria
- Moved `user_prompt` to position 2 in schema to prevent truncation on long responses
- Increased `max_tokens` to 8192 to handle complex multi-part use cases
- Added robust JSON extraction: strips markdown fences and preamble before `{`

### Input validation
- `is_valid_use_case()`: rejects input with < 2 words, < 8 chars, or vowel ratio < 15%
- Catches gibberish like `ncksfnkjcdsn` without an LLM call
- User-facing error message, not developer-facing internals

### Model selection
- Added 6 OpenAI models: `gpt-4o-mini`, `gpt-4o`, `gpt-4.1-nano`, `gpt-4.1-mini`, `gpt-4.1`, `o4-mini`
- `GET /models` endpoint — frontend fetches dynamically, no hardcoding
- Model whitelist validation on backend
- o-series models use `max_completion_tokens` instead of `max_tokens`
- Measured real token usage: ~920 input / ~324 output / ~1,245 total per query

### UI redesign
- Switched from plain CSS to **Tailwind CDN**
- Three visually differentiated output cards:
  - **System Prompt**: dark inverted header (`#18181B`), signals primary importance
  - **User Prompt**: light header, standard treatment
  - **Reviewer Prompt**: dashed border, lighter body, muted text — signals supplementary
- Typography: **Inter** for UI, **JetBrains Mono** for prompt output
- Markdown rendering via `marked.js` — system prompt `##` headings rendered as formatted sections
- Copy button per card with "Copied ✓" feedback
- Empty state hint, hidden after first generation
- Session history (last 10) via `sessionStorage`
- User-friendly error messages — no internals exposed
- Fixed native `<select>` / `<button>` height alignment using `appearance: none`, explicit `height: 38px`, `box-sizing: border-box`, padding removed from both

## Evening session — Refinement loop

### What was built

#### Prompt refinement (`POST /refine`)
- New endpoint in `main.py`: accepts `{system_prompt, user_prompt, feedback, model}`
- Uses `REFINE_PROMPT` as system: instructs LLM to apply feedback precisely, preserve 5-section structure, not weaken constraints
- Same JSON schema + `build_system_prompt()` assembly as `/generate` — returns `GenerateResponse`
- Validates feedback is non-empty; reuses model whitelist and token parameter logic

#### `REFINE_PROMPT` design
- Explicit rules: preserve structure, apply only what feedback requests, no hedging added, user_prompt stays self-contained
- Outputs same 6-key JSON schema as `META_PROMPT` — reuses all existing parsing and assembly code

#### Frontend changes
- Refine section (`#refine-section`) hidden until first generation, then shown below output cards
- Textarea for feedback with descriptive placeholder
- "Refine Prompt" button (outlined secondary style, same height as generate button)
- Iteration badge shows "iteration N" after each refinement, resets on new generate
- Error box per-section for refine failures
- `Cmd+Enter` / `Ctrl+Enter` submits refine if refine input focused, otherwise submits generate

#### UX flow
1. Generate initial prompt
2. Refine section appears
3. Type feedback → Cmd+Enter or click "Refine Prompt"
4. Output cards update in place, iteration counter increments
5. Repeat as many times as needed
6. New generate resets iteration counter to 0
